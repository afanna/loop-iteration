# P1-10 样式值类型系统亲和性评估

## 1. 设计背景

- **协议章节**: 4.2（样式属性）
- **核心问题**: 样式值使用混合类型、统一字符串还是结构化对象？
- **评估动机**: 样式属性的类型不统一可能造成 LLM 生成困惑。当前协议中，样式值使用混合类型：数字（`padding: 16`）、字符串（`backgroundColor: "#FFFFFF"`）、对象（`padding: {"top": 8, "left": 16}`）。需量化评估不同类型系统的模型亲和性。
- **与 V2 协议一致性**: 相同（Same）

## 2. 候选方案

### A: mixed（当前设计，推荐）

数值用数字类型、颜色用字符串、方向性值用对象，按需混合使用：

```json
{"styles": {"padding": 16, "backgroundColor": "#FFFFFF", "borderRadius": {"topLeft": 8, "topRight": 8}}}
```

### B: string-all

所有样式值统一为字符串类型：

```json
{"styles": {"padding": "16", "backgroundColor": "#FFFFFF", "borderRadius": "8 8 0 0"}}
```

### C: structured

所有样式值统一为结构化对象：

```json
{"styles": {"padding": {"value": 16, "unit": "vp"}, "backgroundColor": {"value": "#FFFFFF"}}}
```

## 5. 失败用例分析

### 5.1 失败统计

| 模型 | mixed | string-all | structured |
|------|-------|------------|------------|
| GLM-5.1 | 1 | 1 | 2 |
| DeepSeek-V3 | 0 | 1 | 1 |

### 5.2 代表性失败用例

**GLM mixed（1 个失败）**:

| 用例 ID | 用例名称 | 难度 | 失败级别 | 失败描述 |
|---------|----------|------|----------|----------|
| SV15 | Stack 组件+alignContent+全部样式 | complex | L4 | styles 不包含 "alignContent" |

**GLM string-all（1 个失败）**:

| 用例 ID | 用例名称 | 难度 | 失败级别 | 失败描述 |
|---------|----------|------|----------|----------|
| SV04 | 枚举样式-textAlign | simple | L4 | 字段 value 不存在 — 包含 center |

**GLM structured（2 个失败）**:

| 用例 ID | 用例名称 | 难度 | 失败级别 | 失败描述 |
|---------|----------|------|----------|----------|
| SV03 | 百分比宽度 | simple | L4 | styles 值不包含 "100%" — 包含 100% |
| SV06 | 方向性 padding 对象 | medium | L4 | styles 值不包含 "top" — 包含 top |

**DeepSeek mixed**: 无失败，15/15 全部通过。

**DeepSeek string-all / structured**: 各 1 个失败，模式与 GLM SV04 类似。

### 5.3 失败模式分类

| 失败类型 | mixed (GLM) | string-all (GLM) | structured (GLM) | mixed (DS) | string-all (DS) | structured (DS) |
|----------|-------------|-------------------|-------------------|------------|-----------------|-----------------|
| L1 JSON 解析 | 0 | 0 | 0 | 0 | 0 | 0 |
| L2 字段缺失 | 0 | 0 | 0 | 0 | 0 | 0 |
| L3 模式违反 | 0 | 0 | 0 | 0 | 0 | 0 |
| L4 规则违反 | 1 | 1 | 2 | 0 | 1 | 1 |

所有失败均为 L4 规则违反，无 JSON 解析或结构错误。

### 5.4 根因分析

1. **mixed 的 SV15 失败**: 模型在多属性组合场景中遗漏了 `alignContent` 属性。这是复杂用例的属性完整性问题，与类型系统设计无关。
2. **string-all 的 SV04 失败**: 模型将枚举值 `center` 写为字符串但在字段映射上出错，"所有值都是字符串"反而增加了模型的认知负担。
3. **structured 失败最多**: 结构化对象嵌套增加了认知负担，模型在属性值验证时出现遗漏或格式化错误（如 `100%` 被拆分为 `100` + `%`，`top` 未被正确识别为嵌套属性）。
4. **mixed 在 D1/D2 上明显领先**: DeepSeek mixed D1=100%/D2=100%，string-all 和 structured 均有 1-2 个用例失败。GLM mixed 在 D1/D2 上也优于 string-all/structured。
5. **string-all D4 学习曲线较低**: GLM 0-shot 仅 90%、1-shot 降至 80%，说明"所有值统一为字符串"并非模型天然理解的范式。

### 5.5 优化建议

- mixed 已是最优方案，无需针对类型系统进行优化
- SV15 类型（多属性组合场景）可在 prompt 中增加提示，强调复杂样式属性的完整性
- 如需进一步提升 GLM 在复杂用例上的表现，可考虑拆分多属性组合为多个生成步骤

## 6. 结论

- **推荐方案**: mixed（当前设计）胜出，MA 均值 98.3%（A+ 级）
- **与 V2 协议一致性**: 相同（当前设计即 V2 协议设计）
- **风险提示**: 无

核心发现：

1. mixed 在两个模型中均表现最优或并列最优：GLM 96.7% (A+) / DeepSeek 100.0%（A+），且两个模型均达到 A+ 级别
2. string-all 与 structured 性能接近：DeepSeek 上两者完全一致（96.7%），GLM 上 string-all 略优（95.6% vs 93.4%）
3. mixed 在 D1/D2 上明显领先：DeepSeek mixed D1=100%/D2=100%，说明混合类型最接近 LLM 在主流 UI 框架（CSS/Flexbox）训练数据中习得的模式
4. structured 失败最多（GLM 2 个），结构化对象嵌套过深增加模型认知负担
5. 三个方案全部达到 A 级以上，说明样式值类型系统的整体模型亲和性良好

**结论**: 保持 A（mixed）方案。当前协议的混合类型设计（数字/字符串/对象按需使用）是最优选择 — 模型能自然区分不同属性的类型，混合类型不影响生成准确率，且是最接近主流 UI 框架的模式，LLM 训练数据中天然熟悉。

```bash
ONLY_MODEL=glm npm run eval:style-type

# 仅运行 DeepSeek
ONLY_MODEL=deepseek npm run eval:style-type
```
