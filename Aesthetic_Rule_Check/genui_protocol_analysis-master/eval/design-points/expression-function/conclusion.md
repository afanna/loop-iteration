## 设计点 #2：表达式包装方式

### 1. 设计背景

- **协议章节**: 3.5（表达式）
- **核心问题**: 动态值用 `{{ }}` 内联还是 `{"expr":}` 对象包装？
- **评估动机**: 表达式包装方式是最基础的设计决策，影响所有其他表达式相关设计

### 2. 候选方案

**A: inline（推荐）**
```json
{"content": "{{ $user.name }}", "styles": {"fontSize": "{{ $size + 2 }}"}}
```

**B: wrapped**
```json
{"content": {"expr": "$user.name"}, "styles": {"fontSize": {"expr": "$size + 2"}}}
```

两种方案在表达式语言层面完全相同（变量 `$var`、运算符、函数直接调用），区别仅在包装方式。

### 3. 测试用例

- **文件**: `test-cases/expression-wrap.json`
- **总计**: 20 个用例
- **层级**: 简单绑定 → 混合文本+函数 → 运算符 → 复杂组合

### 5. 失败用例分析

**两种方案均无测试用例失败**（所有 20 个用例全部通过）。

MA 差异仅 ±0.1%，在统计噪声范围内。D6 结构一致率上 wrapped 稍优，但差异来自 `id` 字段的有无，与包装方式本身无关。

### 6. 结论

**推荐 inline `{{ }}` 方案**，理由：
1. 客户端实现更简洁统一：所有动态值都是 string 类型
2. 无类型歧义：`{{ }}` 字符串不可能与普通值混淆；wrapped 方案中 `{expr:}` 对象与普通对象可能混淆
3. 模型亲和性等价（MA 差异 ±0.1%）

## 设计点 #14：表达式内置函数

### 1. 设计背景

- **协议章节**: 4.4.5（内置函数）
- **核心问题**: 字符串格式化使用 `format()` 函数还是模板字符串？两者共存是否影响一致性？
- **评估动机**: 格式化是最高频的表达式操作之一

### 2. 候选方案

**A: format（槽位式）**
```json
{"content": "{{ format('Hello, {}!', $name) }}"}
{"content": "{{ format('{} 有 {} 条消息', $userName, $messageCount) }}"}
```

**B: template-literal（模板字符串）**
```json
{"content": "{{ `Hello, ${$name}!` }}"}
{"content": "{{ `${$userName} 有 ${$messageCount} 条消息` }}"}
```

**C: coexist（两者共存, 推荐）**
```json
// 两种方式均可使用
{"content": "{{ format('共{}件', $count) }}"}       // format 适合多参数
{"content": "{{ `Hello, ${$name}!` }}"}              // 模板字符串适合直观插值
```

#### 失败模式分类统计

| 失败类型 | format (GLM) | template (GLM) | coexist (GLM) | format (DS) | template (DS) | coexist (DS) |
|---------|-------------|----------------|---------------|-------------|---------------|--------------|
| L1 JSON 解析失败 | 0 | 0 | 0 | 0 | 0 | 0 |
| L2 结构/字段缺失 | 0 | 0 | 0 | 0 | 0 | 0 |
| L3 模式不匹配 | 0 | 0 | 0 | 0 | 0 | 0 |
| L4 规则违反 | 0 | 0 | 1 | 0 | 0 | 1 |

#### 根因分析

1. **coexist 唯一失败点是 BF20**：该用例要求在多组件场景中，子组件使用特定的格式化方式。共存方案下模型在两个子组件中可能混合使用不同方式，导致不符合特定策略的验证规则
2. **GLM coexist D6 语义等价率仅 13%**：共存方案导致模型在不同次生成中随机选择 format 或 template-literal，语义等价性大幅下降。这是共存方案的固有代价——模型面临选择歧义
3. **format 和 template-literal 单一方案 D1/D2 均为 100%**：LLM 对两种方式都能完美理解和生成

#### 对胜出策略（coexist）的优化建议

1. **在 prompt 中明确推荐优先使用 template-literal**，减少选择歧义
2. **BF20 类用例的验证规则可放宽**：只要求每个子组件使用其中一种方式即可，不强制统一
3. **考虑在协议规范中明确两种方式的使用场景指引**：format 用于精确格式控制，template-literal 用于直观插值

### 6. 结论

**推荐方案：两者共存，以 template-literal 为主、format 为辅**

| 方案 | GLM MA | DeepSeek MA | 平均 MA | 0-shot | 推荐度 |
|------|--------|-------------|---------|--------|-------|
| format | 96.4% | 99.5% | 98.0% | 100% | 可选 |
| template-literal | 97.2% | 99.2% | 98.2% | 100% | 主推 |
| coexist | 90.2% | 97.6% | 93.9% | 90% | 推荐 |

理由：
1. format 和 template-literal 无显著差异（平均 MA 差值仅 0.3%）
2. 实际协议需要两种方式配合使用
3. coexist 的 MA 下降主要由 D6（一致性）导致，不影响实际生成质量
4. template-literal 更接近 JS/TS 语法，LLM 训练数据覆盖更广

- **V2 协议一致性**: 相同

## 附录：类型转换 `+` 运算符亲和性分析

对协议中 `+` 运算符的不对称类型转换规则（`string + number` → 拼接，`number + string` → 尝试数值解析）进行了量化验证：
