# #12 表达式运算符范围 — 亲和性评估

## 1. 设计背景

- **协议章节**: 4.4.3（运算符集合）
- **核心问题**: 运算符集合大小对 LLM 生成准确率的影响？
- **评估动机**: 运算符越多选择越多出错概率越高，但精简子集需用函数替代
- **与 V2 协议一致性**: 相同（Same）

协议 4.4.3 节定义了表达式的运算符集合。运算符范围的大小直接影响 LLM 生成表达式时的准确率（运算符越多，选择越多，出错概率越高），同时影响协议的实现复杂度（全量运算符需要实现完整的优先级解析）和语义歧义风险（如 `+` 既是算术加法又是字符串拼接）。本评估对比三种运算符范围策略，量化模型对每种策略的亲和性差异。

## 2. 候选方案

### A: full（全量运算符，推荐）

```
算术: + - * / %
比较: == != < > <= >=
逻辑: && || !
三元: ? :
共 10 级优先级
```

LLM 可直接使用运算符组合复杂表达式，表达自然，开发者认知负担低。

### B: subset（精简子集）

```
比较: == != < > <= >=
逻辑: && || !
三元: ? :
字符串拼接: +
算术通过函数: multiply(a,b), divide(a,b), add(a,b), subtract(a,b), modulo(a,b)
```

减少运算符种类，降低优先级解析复杂度，算术运算通过函数替代。

### C: ternary（仅三元表达式）

```
三元: ? :
字符串拼接: +
比较/逻辑/算术全部通过函数: eq(), neq(), gt(), lt(), gte(), lte(), and(), or(), not(), add(), multiply()...
```

运算符最少，优先级最简单，所有比较和逻辑运算均通过函数调用实现。

## 5. 失败用例分析

### 5.1 GLM-5.1 失败情况

**无失败** — 三种策略下 16 个用例全部通过。

### 5.2 DeepSeek-V3 full（1 个失败）

| 用例 ID | 用例名称 | 复杂度 | 失败级别 | 失败描述 |
|---------|----------|--------|----------|----------|
| OS04 | 逻辑NOT三元 | simple | L4 | content 不包含 `!`，模型用 `$isHidden ? '已隐藏' : '可见'` 而非 `!$isHidden ? '可见' : '已隐藏'` |

**失败模式**: 模型避免使用 `!` 运算符，选择直接调换三元表达式分支来规避逻辑 NOT 操作。这是模型的行为偏好，而非协议设计缺陷。

### 5.3 DeepSeek-V3 subset（1 个失败）

| 用例 ID | 用例名称 | 复杂度 | 失败级别 | 失败描述 |
|---------|----------|--------|----------|----------|
| OS04 | 逻辑NOT三元 | simple | L4 | 与 full 策略相同的模式 — 模型避免使用 `!` 运算符 |

### 5.4 DeepSeek-V3 ternary（0 个失败）

全部通过。

### 5.5 失败模式对比总结

| 失败类型 | full (GLM) | subset (GLM) | ternary (GLM) | full (DS) | subset (DS) | ternary (DS) |
|----------|-----------|-------------|---------------|-----------|-------------|--------------|
| L4 规则违反 | 0 | 0 | 0 | 1 | 1 | 0 |

### 5.6 根因分析

1. **DeepSeek 避免 `!` 运算符** — 模型倾向于调换三元表达式分支来规避逻辑 NOT 操作（用 `$isHidden ? '已隐藏' : '可见'` 替代 `!$isHidden ? '可见' : '已隐藏'`），这是模型的行为模式，非协议设计问题。
2. **三种策略差异极小** — MA 综合分差在 3% 以内，运算符范围对模型亲和性的影响非常有限。
3. **ternary 策略在 DeepSeek 上得分最高** — 函数式调用无需额外示例即可正确使用（D4 学习曲线 0-shot 即达 100%），但 GLM 上 full 策略更稳定（D6 一致性最高）。

### 5.7 优化建议

- 推荐使用 full 全量运算符，模型对 `!` 运算符的行为偏好可通过 prompt 工程优化解决
- 如需降低实现复杂度，可考虑 subset 方案：仅移除算术运算符用函数替代，对 LLM 准确率几乎无影响
- 运算符范围对模型亲和性的整体影响有限，优先级低于其他设计点

## 6. 结论

- **推荐方案**: full（全量运算符）
- **与 V2 协议一致性**: 相同
- **风险评估**: 无 — 三种策略 MA 差异 <3%，均达到生产可用标准（A 级或以上）

推荐 full 全量运算符的理由：
1. GLM-5.1 作为主要目标模型，full 策略得分最高（98.0% A+）
2. 全量运算符表达更自然，开发者认知负担低
3. 实现端增加的优先级解析成本有限
4. 三种策略差异极小（<3%），full 在主要维度无劣势

如需降低实现复杂度，可考虑方案 B（subset）：仅移除算术运算符用函数替代，对 LLM 准确率几乎无影响。
