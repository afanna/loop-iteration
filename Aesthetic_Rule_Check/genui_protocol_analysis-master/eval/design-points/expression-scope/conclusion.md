# #13 表达式使用范围约束

## 1. 设计背景

- **协议章节**: 3.5（表达式）
- **核心问题**: 表达式范围约束（仅扩展组件、仅值位置等 7 条规则）LLM 能否自然遵守？
- **评估动机**: 如果 LLM 经常在禁止字段使用表达式，说明约束设计需要调整。表达式使用范围约束直接影响 LLM 生成 UI DSL 的准确性——模型必须理解哪些字段可用表达式、哪些字段必须固定。

## 2. 候选方案

### A: 当前设计（推荐）

表达式仅在 `Extended.*` 组件的属性值位置可用，禁止在 id/component/theme/label/children/action 等 7 类字段使用。7 条约束全部生效。

### B: 所有组件

取消"仅扩展组件"约束，所有组件的属性值都可以用表达式。仅保留约束 2-7。

### C: 包括 key

表达式可以在任何位置使用，包括 JSON key 位置。取消约束 1 和 6。

## 5. 失败用例分析

### 5.1 GLM-5.1（3 个失败）

| 用例 ID | 用例名称 | 复杂度 | 失败类型 |
|---------|----------|--------|----------|
| ES13 | 综合-混合可表达式和禁止字段 | complex | L1 JSON 解析失败 |
| ES19 | 嵌套组件-正确边界 | complex | L1 JSON 解析失败 |
| ES20 | 综合场景-购物车 | complex | L1 JSON 解析失败 |

**失败分类**: 3 个 L1 JSON 解析失败，全部集中在 complex 复杂度场景。

### 5.2 DeepSeek-V3（1 个失败）

| 用例 ID | 用例名称 | 复杂度 | 失败类型 |
|---------|----------|--------|----------|
| ES13 | 综合-混合可表达式和禁止字段 | complex | L4 — children 不应包含 `{{`（children 不能用表达式） |

**失败分类**: 1 个 L4 规则违反——模型在 children 字段使用了表达式。

### 5.3 失败类型统计

| 失败类型 | GLM | DeepSeek |
|----------|-----|----------|
| L1 JSON 解析 | 3 | 0 |
| L2 字段缺失 | 0 | 0 |
| L3 规则违反 | 0 | 0 |
| L4 规则违反 | 0 | 1 |

### 5.4 根因分析

1. **GLM 失败全部是输出截断** — 3 个 complex 用例因输出 JSON 过长超出 token 限制导致解析失败，并非范围约束违反。本质上是生成能力问题，而非协议理解问题。
2. **DeepSeek ES13 是唯一的约束违反** — 模型在任务描述过于复杂时注意力分散，在 children 字段使用了表达式。这是唯一的实质性范围约束违反。
3. **所有 simple/medium 用例全部通过** — 两个模型在 ES01-ES12（简单和中等复杂度）中全部通过所有范围约束，说明 `component`、`id`、`theme`、`label`、`placeholder`、`children`、`action` 不可用表达式的规则与 LLM 训练数据中的 JSON/编程常识高度一致。
4. **D4 学习曲线完全平坦** — 0-shot 即达 100%，零学习成本。LLM 无需示例即可正确遵守范围约束。
5. **D6 一致性 100%** — 范围约束理解是确定性的，不是"偶尔记起"的规则。

### 5.5 优化建议

- 在协议规范中明确列出表达式可用/不可用的字段白名单/黑名单
- 对于特别复杂的嵌套场景，考虑在 catalog 中进一步细化约束
- GLM 复杂场景的 JSON 截断问题属于模型工程限制，可通过拆分生成步骤缓解

## 6. 结论

- **推荐方案**: 维持当前设计（方案 A）
- **与 V2 协议一致性**: 相同
- **风险等级**: 低 — 核心约束被 LLM 自然理解

**推荐理由**:

1. 核心约束（禁止在 id/component/theme/label/children/action 中使用表达式）与 LLM 训练数据常识一致
2. 0-shot 即可完美遵守，零学习成本（D4 学习曲线完全平坦）
3. D6 一致性 100% 证明约束理解是确定性的
4. DeepSeek-V3 达到 A+ 级（96.6%），GLM-5.1 为 A 级（87.0%，主要失分在复杂场景输出能力而非约束理解）
