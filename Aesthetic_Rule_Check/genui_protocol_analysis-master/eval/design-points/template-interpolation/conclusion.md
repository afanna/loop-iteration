# 模板字符串求值语法 — 模型亲和性评估

## 1. 设计背景

- **协议章节**: §4.4.2（数据类型）, §4.4.5（内置扩展函数）, §4.4.8（EBNF）
- **核心问题**: 模板字符串中变量引用和表达式求值的语法规则是什么？`$var` 和 `${expr}` 如何分工？
- **评估动机**: #16 和 #20 混淆了"变量引用方式"和"模板字符串语法"两个不同层面的问题。需要建立清晰的设计规则。

## 2. 设计规则

1. **`$var`** — 变量的唯一引用形式，在表达式和模板字符串中通用
2. **`${ expr }`** — 模板字符串内的表达式求值语法，对 expr 求值并转换为字符串参与拼接

## 3. 候选方案

| 策略 | 模板中 `$var` | 模板中 `${expr}` | 说明 |
|------|:--:|:--:|------|
| **A: template-text** | 插值 | 插值 | `$var` 可在模板文本中直接使用；`${}` 用于嵌入复杂表达式 |
| **B: explicit-only** | 字面量 | 插值 | 模板中所有插值必须通过 `${}`，裸 `$var` 为原样文本 |

### 策略 A 示例
```jsonc
{"content": "{{ `Hello, $name!` }}"}                        // $var 直接嵌入
{"content": "{{ `Hello, ${$name}!` }}"}                      // ${} 包裹，等价
{"content": "{{ `Total: ${$price * $quantity}` }}"}          // ${} 嵌入算术表达式
{"content": "{{ `共 ${size($items)} 件` }}"}                 // ${} 嵌入函数调用
{"content": "{{ `$userName has ${$messageCount} msgs` }}"}   // 混合
```

### 策略 B 示例
```jsonc
{"content": "{{ `Hello, ${$name}!` }}"}                      // 必须 ${}
{"content": "{{ `Total: ${$price * $quantity}` }}"}          // ${} 嵌入表达式
// `Hello, $name!` → 渲染为 "Hello, $name!"（字面量），策略B视为错误
```

### 失败用例分析

#### 策略 A (template-text)

**双模型均失败 1 例：TI14 多组件多字段动态文本** — children 字段中 size() 未放在 ${} 内。这是多组件 children 序列化导致的正则匹配问题，不是模板语法错误。

#### 策略 B (explicit-only)

**双模型均失败 1 例：TI06 混合$var和${expr}** — explicit-only 策略要求所有插值必须用 ${}，但模型在混合场景中仍然在 ${} 外使用了 $userName。说明 $var 在模板文本中的"插值直觉"很强，强制禁止反而导致混淆。

## 6. 结论

**推荐策略 A: `$var` 直接嵌入 + `${expr}` 求值**

| 方案 | GLM MA | DeepSeek MA | 平均 MA | 推荐度 |
|------|:--:|:--:|:--:|:--:|
| **A: template-text** | **93.2% (A+)** | **95.7% (A+)** | **94.4%** | **推荐** |
| B: explicit-only | 92.2% (A+) | 94.2% (A+) | 93.2% | 备选 |

理由：
1. **策略 A 双模型均领先** — GLM +1.0%, DeepSeek +1.5%
2. **D4 0-shot 100%** — 策略 A 双模型 0-shot 均为 100%，无需示例即可正确生成；策略 B 仅 90%
3. **DeepSeek 策略 A MA 95.7%** — 超过 95% 阈值，达到生产级亲和性
4. **LLM 天然倾向 $var 模板插值** — explicit-only 强制禁止裸 $var 导致 TI06 失败（将 `$userName` 放在 `${}` 外被验证器拒绝）
