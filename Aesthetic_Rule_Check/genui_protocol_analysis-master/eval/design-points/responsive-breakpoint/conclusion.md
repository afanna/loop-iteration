# #21 响应式断点表达方式亲和性评估 — `$__WidthBreakpoint` vs `$__WindowSize.width`

## 1. 设计背景

- **协议章节**: 3.6（多端设备适配）
- **核心问题**: 响应式 DSL 中，使用断点枚举字符串比较（`$__WidthBreakpoint == 'sm'`）还是窗口宽度数值比较（`$__WindowSize.width < 768`）更模型亲和？
- **评估动机**: 协议综合示例（3.6.4）两种方式混用，需要确定哪种应作为推荐方式

## 2. 候选方案

| 场景    | A: breakpoint-enum                                                 | B: window-size                                                    |
| ----- | ------------------------------------------------------------------ | ----------------------------------------------------------------- |
| 条件样式  | `"fontSize": "{{ $__WidthBreakpoint == 'sm' ? 14 : 16 }}"`        | `"fontSize": "{{ $__WindowSize.width < 768 ? 14 : 16 }}"`         |
| IF 条件 | `"condition": "{{ $__WidthBreakpoint == 'sm' }}"`                 | `"condition": "{{ $__WindowSize.width < 768 }}"`                  |
| 多断点   | `$bp == 'sm' ? a : $bp == 'md' ? b : c`                            | `$w < 576 ? a : $w < 992 ? b : c`                                 |
| 显示/隐藏 | `"display": "{{ $__WidthBreakpoint != 'sm' ? 'flex' : 'none' }}"` | `"display": "{{ $__WindowSize.width >= 768 ? 'flex' : 'none' }}"` |

**关键差异**：
- A 语义清晰（`'sm'` = 手机），无魔数，但仅限预定义断点
- B 灵活（任意像素阈值），接近 CSS media query，但需模型记住断点边界值

## 3. 测试用例

- **文件**: `test-cases/responsive-breakpoint.json`
- **总计**: 15 个（simple 5 / medium 5 / complex 5）
- **边界用例**: 6 个

覆盖场景：响应式间距、字体、显示隐藏、文本切换、对齐、布局方向切换、导航栏切换、单列/双列、侧边栏显隐、卡片样式、三断点字体、三断点布局、完整响应式页面、响应式网格、混合业务页面。

## 5. 失败用例分析

### 优化后全部通过（0 个失败）

通过以下两个优化措施，所有失败用例已消除：
1. **增加多组件数组输出的 few-shot 示例**：展示 Extended.If + 展开子组件的 JSON 数组格式
2. **task 描述中明确要求"请输出JSON数组，展开所有子组件的完整定义"**
3. **RB14 task 中明确指定 columnsGap/rowsGap 属性名称**

### 优化前失败用例（已修复）

1. **RB06 布局方向切换**（DeepSeek breakpoint-enum）：输出了 If 组件但未展开 Column/Row 分支组件。根因是 few-shot 示例只有单组件输出，模型不确定是否需要展开子组件
2. **RB14 响应式网格列数**（DeepSeek breakpoint-enum）：使用了 `space` 而非 `columnsGap` 属性名。根因是 task 描述中"行间距和列间距都为16"未指定具体属性名

## 6. 结论

- **推荐方案**: 两方案均可 — 亲和性无显著差异
- **V2 协议一致性**: 维持当前两种方式共存的设计（协议示例已使用两种方式）
- **风险**: 无，两种方案均达到 A+ 级

| 方案 | GLM MA | DeepSeek MA | 平均 MA | 0-shot | 推荐度 |
|------|--------|-------------|---------|--------|-------|
| A: breakpoint-enum `$__WidthBreakpoint` | 96.7% (A+) | 98.7% (A+) | 97.7% | 90% | **推荐** |
| B: window-size `$__WindowSize.width` | 97.7% (A+) | 98.0% (A+) | 97.9% | 90% | **推荐** |

### 附加建议

1. **两种方式亲和性一致**：优化 few-shot 示例后，两种策略 MA 差异仅 0.3-0.7%，模型对断点枚举和数值比较同等亲和
2. **breakpoint-enum 语义更清晰**：`$__WidthBreakpoint == 'sm'` 比数值比较更直观，适合 DSL 可读性
3. **window-size 灵活性更高**：可使用任意像素阈值，适合非标准断点场景
4. **多组件输出需要明确引导**：通过 few-shot 示例展示数组格式 + task 描述明确要求展开子组件，可显著提升 D6 一致性和 D5 边界鲁棒性
5. **0-shot 学习能力好**：两种方案 0-shot 准确率均达 90%+，说明无需额外示例模型即可正确使用
