# GAP-023: 表达式无 `{{` 字面量转义机制

## 问题描述

协议 3.5 节定义了表达式语法 `{{<expression>}}`，但没有定义如何在文本中输出字面量 `{{` 或 `}}`。

场景：UI 需要显示文本 "价格为 {{price}} 元"（不是表达式，是用户可见的字面量文本）。

当前协议下 LLM 无法处理此场景：
- 如果直接写 `"text": "价格为 {{price}} 元"`，客户端会把它当表达式解析
- 如果写 `"text": "{{ '价格为 {{price}} 元' }}"`，内层 `{{` 违反"不支持嵌套表达式"的约束

## 影响范围

- 协议章节: 3.5（表达式）、4.4.1（基本语法）
- 测试分类: FP-04 (expression)，需新增
- 严重性: 这是功能缺失（missing feature），不是 bug。当前协议下某些 UI 文本根本无法表达。

## 候选修复方案

- **方案 A**: 新增转义语法，如 `\{{` 表示字面量 `{{`
  - 示例: `"text": "价格为 \\{{ price \\}} 元"` → 渲染为 "价格为 {{price}} 元"
  
- **方案 B**: 使用 Unicode 转义
  - 示例: `"text": "价格为 \u007B\u007Bprice\u007D\u007D 元"`
  
- **方案 C**: 引入 `#` 作为禁用表达式前缀（协议 4.4.7 已预留 `#` 为保留字符）
  - 示例: `"text": "#价格为 {{price}} 元"` → `#` 禁用表达式解析

- **推荐**: 方案 C。利用已有的 `#` 保留字符，不引入新语法，改动最小。

## 验证计划

**类型**: 实质性修复 — affinity-design A/B 对比验证

1. 在 `eval/design-points/` 新建 `expression-escape/` 设计点
2. 测试用例：需要 LLM 生成包含字面量 `{{` 的 UI 文本
3. A/B 对比：
   - 策略 A（无转义机制）：LLM 只能回避或输出错误
   - 策略 B（`#` 前缀禁用）：LLM 需要学会在需要时加 `#`
4. 如果 0-shot 通过率低 → 补充 few-shot 示例
5. 策略 B 达到 A 级 → 修改协议
