# 模型亲和性量化评估模型（MA-6D）

## 1. 概述

模型亲和性量化评估模型（Model Affinity - 6 Dimensions, MA-6D）用于度量 LLM 对 UI 协议 DSL 的生成能力。其核心思路是：**一个好的协议设计，应当让模型无需特殊训练即可准确、高效、稳定地生成符合规范的 DSL。**

MA-6D 定义了 6 个正交的评估维度，每个维度度量生成能力的一个独立侧面，最终加权合成为综合评分 MA（Model Affinity Score），并映射到 A+/A/B/C/D 五个等级。

### 评估目标

| 层次 | 目标 | 说明 |
|------|------|------|
| **造尺子** | 构建量化评估框架 | 6 维评分 + S/A/B/C/D 分级，产出可复用的评估工具链 |
| **量单点** | 单点设计对比评估 | 针对某个设计点的多个候选方案，用量化评估选择模型最亲和的设计 |
| **做验收** | 整体协议验收评估 | 对完整协议进行端到端的模型亲和性量化评估，判断是否达到生产可用标准 |

---

## 2. 评估流程

```
测试用例 → Prompt 构建 → LLM 生成 → 分层验证 → 6 维评分 → MA 综合评分
              ↑                          ↓
              └── 策略感知（可选）←── 策略特定验证规则
```

### 2.1 测试用例

每个测试用例包含：

| 字段 | 说明 |
|------|------|
| `id` / `name` | 唯一标识和名称 |
| `category` | 分类：expression / event / layout / component / mixed / variable-scope / event-chain 等 |
| `complexity` | 复杂度：simple / medium / complex |
| `task` | 给 LLM 的自然语言任务描述 |
| `expected` | 期望输出结构（component 类型、必选字段、模式匹配规则） |
| `validation_rules` | 验证规则列表 |
| `hints`（可选） | 协议提示，注入到 prompt 中辅助生成 |

策略感知用例（如 `VariableScopeTestCase`、`ChainTestCase`）额外包含：

- `shared_rules`：所有策略共用的验证规则
- `strategy_rules`：按策略区分的验证规则（如 shadowing vs explicit）
- `scope_conflict_type` / `chain_depth` 等策略特定参数

### 2.2 分层验证（L1-L4）

LLM 的每次输出经过 4 级递进验证，任何一级失败都意味着后续级别自动失败：

| 级别 | 名称 | 检查内容 | 失败示例 |
|------|------|----------|----------|
| **L1** | JSON 解析 | 能否从输出中提取并解析出有效 JSON | 输出包含 markdown 标记、不是 JSON |
| **L2** | 结构验证 | 必选字段是否存在、component 类型是否正确 | 缺少 `component` 字段、`component` 不是期望值 |
| **L3** | 模式匹配 | 字段值是否包含指定子串、是否匹配正则、类型是否正确 | 变量引用缺少 `$` 前缀、字段类型错误 |
| **L4** | 规则验证 | 策略感知的业务规则是否全部通过 | 变量引用方式不符合策略、事件链格式错误 |

L4 规则验证支持以下规则类型：

- `has_field` — 字段必须存在
- `contains` / `not_contains` — 字段值包含/不包含指定内容
- `matches_regex` — 字段值匹配正则
- `field_is_expr` — 字段必须使用 `{expr: "..."}` 格式
- `listeners_format` — 事件监听器格式校验
- `children_format` — 子组件格式校验
- `chain_format` — 事件链格式校验（区分 handlerGroups / flat-array 策略）

### 2.3 Prompt 构建

System Prompt 由协议摘要 + 基础规则 + 策略特定规则组成。User Prompt 由 few-shot 示例 + 任务描述 + 验证规则 + 提示信息组成。

Few-shot 示例按 category 分组，支持 0/1/3 shot 可变数量，用于 D4 学习曲线评估。

## 3. 六维评分模型

### 3.1 维度总览

| 维度 | 名称 | 权重 | 度量目标 | 核心问题 |
|------|------|------|----------|----------|
| **D1** | 语法准确率 | 20% | 输出格式正确性 | 模型能生成格式正确的 DSL 吗？ |
| **D2** | 语义准确率 | 25% | 业务语义正确性 | 模型能正确理解并实现业务意图吗？ |
| **D3** | 生成效率 | 15% | 资源消耗和首次成功率 | 模型需要多少次尝试和 token 才能生成正确结果？ |
| **D4** | 概念收敛速度 | 15% | 协议可学习性 | 给几个示例后模型能多快掌握协议概念？ |
| **D5** | 边界鲁棒性 | 15% | 极端场景处理能力 | 面对复杂/冲突/极端用例模型还能正确生成吗？ |
| **D6** | 一致稳定性 | 10% | 输出稳定性 | 多次生成同一任务，结果一致吗？ |

### 3.2 D1 — 语法准确率（Syntactic Accuracy）

**度量目标**：模型输出的 DSL 在格式层面是否正确。

**计算方法**：基于 L1-L4 四级验证的加权通过率。

```
D1 = (L1通过率 × 0.2) + (L2通过率 × 0.3) + (L3通过率 × 0.3) + (L4通过率 × 0.2)
```

其中 Lx通过率 = 通过 Lx 级别验证的用例数 / 总用例数。

**权重设计理由**：
- L2（结构）和 L3（模式）权重较高（各 0.3），因为它们反映了协议核心语法的正确性
- L1（JSON 解析）是基本门槛，权重较低（0.2）
- L4（规则）虽然重要，但在 D2 中作为语义准确率的主要度量，此处权重也设为 0.2 以避免重复计算

**评分范围**：[0, 1]，1 表示所有用例在所有级别都通过。

### 3.3 D2 — 语义准确率（Semantic Accuracy）

**度量目标**：模型生成的 DSL 是否在业务语义上完全正确，即满足协议的所有设计约束。

**计算方法**：L4 规则全部通过的用例比例。

```
D2 = L4全部通过的用例数 / 总用例数
```

**设计理由**：
- L4 验证是策略感知的，会根据不同设计策略应用不同的验证规则
- 只有 L4 通过，才说明模型真正理解了协议的语义约束
- D2 的权重最高（25%），因为语义正确性是协议设计最关键的指标

**评分范围**：[0, 1]，1 表示所有用例的语义规则全部通过。

### 3.4 D3 — 生成效率（Generation Efficiency）

**度量目标**：模型生成正确 DSL 的资源消耗和效率。

**计算方法**：综合首次通过率、平均重试次数和平均 token 消耗。

```
D3 = (首次通过率 × 0.5) + (重试评分 × 0.3) + (token评分 × 0.2)
```

其中：
- **首次通过率** = retries=0 的用例数 / 总用例数
- **重试评分** = 1 - min(平均重试次数 / maxRetries, 1)
- **token评分** = 1 - min(平均token数 / tokenBaseline, 1)
  - `maxRetries` 默认为 1，`tokenBaseline` 默认为 500

**权重设计理由**：
- 首次通过率权重最高（0.5），因为生产环境中通常不提供重试机会
- 重试评分（0.3）反映容错成本
- token 评分（0.2）反映计算资源消耗

**评分范围**：[0, 1]，1 表示所有用例首次通过且 token 消耗极低。

### 3.5 D4 — 概念收敛速度（Learning Curve）

**度量目标**：协议概念的"可学习性"——模型在获得不同数量的示例后，能多快掌握协议。

**计算方法**：通过控制 few-shot 示例数量（0-shot、1-shot、3-shot），分别测量准确率。

```
D4 = (shot0准确率 × 0.3) + (delta_0_to_1 × 0.3) + (shot3准确率 × 0.4)
```

其中：
- **shot0 准确率** = 0-shot（无示例）时的通过率
- **delta_0_to_1** = max(shot1准确率 - shot0准确率, 0)，即 1 个示例带来的提升幅度
- **shot3 准确率** = 3-shot（3 个示例）时的通过率

**权重设计理由**：
- shot3 准确率权重最高（0.4），代表充分示例后的最终掌握程度
- delta_0_to_1（0.3）反映单个示例的边际收益——好的协议应该一个示例就能让模型理解核心概念
- shot0 准确率（0.3）反映协议本身的直觉性——好的协议应该让模型零示例就能猜对一部分

**评分范围**：[0, 1]，1 表示 0-shot 即完美通过，且示例无额外增益。

**实施方式**：对同一批测试用例分别用 0/1/3 个 few-shot 示例运行，收集三组准确率数据。

### 3.6 D5 — 边界鲁棒性（Edge Robustness）

**度量目标**：模型在面对复杂、冲突、极端等边界场景时的生成质量。

**计算方法**：边界用例的通过率。

```
D5 = 边界用例通过数 / 边界用例总数
```

**边界用例类型**：
- 变量名冲突（`scope_conflict_type: "shadowing"` / `"triple"`）
- 多层嵌套引用
- 复杂条件表达式
- 多步事件链（chain_depth ≥ 3）
- 类型转换

**设计理由**：
- 正常用例的高通过率可能掩盖边界问题
- 生产环境中边界场景是主要的故障来源
- 独立度量边界鲁棒性有助于发现协议设计中的脆弱点

**评分范围**：[0, 1]，1 表示所有边界用例全部通过。

### 3.7 D6 — 一致稳定性（Consistency Stability）

**度量目标**：模型对同一任务多次生成的输出是否一致。

**计算方法**：对同一用例重复生成 N 次，比较所有输出对的结构一致性和语义等价性。

```
D6 = (结构一致率 × 0.4) + (语义等价率 × 0.6)
```

其中：
- **结构一致率** = 所有输出的 JSON key 结构完全相同的用例比例
- **语义等价率** = 所有输出在关键字段（component、label、content、listeners 等）上值相同的用例比例

**结构一致性比较**：递归比较 JSON 的 key 层级和类型结构，忽略具体值。

**语义等价性比较**：
1. `component` 字段必须一致
2. 关键字段（label、content、placeholder、theme、text、condition、isOn、value、src）值必须一致
3. `listeners` 中的事件类型集合必须一致

**权重设计理由**：
- 语义等价权重更高（0.6），因为值相同比结构相同更重要
- 结构一致（0.4）反映模型输出格式的稳定性

**评分范围**：[0, 1]，1 表示所有用例多次生成的输出在结构和语义上完全一致。

## 4. 综合评分 MA

### 4.1 计算公式

```
MA = D1 × 0.20 + D2 × 0.25 + D3 × 0.15 + D4 × 0.15 + D5 × 0.15 + D6 × 0.10
```

### 4.2 权重设计理由

| 维度 | 权重 | 理由 |
|------|------|------|
| D2 语义准确率 | **25%** | 最高权重。语义正确性是协议最核心的价值——生成的 DSL 必须在业务含义上正确 |
| D1 语法准确率 | **20%** | 第二权重。格式正确是语义正确的前提，但相对容易通过重试修复 |
| D3 生成效率 | **15%** | 重要但不决定性。生产环境中可以通过缓存、预计算等手段缓解 |
| D4 概念收敛 | **15%** | 重要但不决定性。通过 few-shot 可以持续改善，但反映协议的内在质量 |
| D5 边界鲁棒性 | **15%** | 重要但不决定性。边界场景在总用例中占比较小，但影响系统可靠性 |
| D6 一致稳定性 | **10%** | 最低权重。一致性影响用户体验，但可以通过确定性采样策略缓解 |

### 4.3 等级映射

| 等级 | 分数范围 | 含义 |
|------|----------|------|
| **A+** | [0.90, 1.00] | 卓越 — 协议设计对模型高度亲和，可直接用于生产环境 |
| **A** | [0.80, 0.90) | 优秀 — 协议设计基本就绪，少量优化后可用于生产 |
| **B** | [0.70, 0.80) | 良好 — 协议设计方向正确，但存在明显短板需要改进 |
| **C** | [0.60, 0.70) | 及格 — 协议设计有较多问题，需要重点改进 |
| **D** | [0, 0.60) | 不合格 — 协议设计需要重新考虑 |

**验收标准**：等级 A 及以上表示协议的模型亲和性达到生产可用标准。

## 5. 使用场景

### 5.1 单点设计对比评估

当协议的某个设计点存在多个候选方案时（例如变量引用采用 shadowing 还是 explicit 策略），对每个方案分别运行 MA-6D 评估，通过对比各维度得分选择最优方案。

```
方案 A (shadowing):  D1=0.85  D2=0.78  D3=0.90  D4=0.72  D5=0.65  D6=0.88  → MA=0.79 (B)
方案 B (explicit):   D1=0.88  D2=0.85  D3=0.85  D4=0.80  D5=0.75  D6=0.82  → MA=0.83 (A)
```

### 5.2 整体协议验收评估

协议设计完成后，用完整的测试用例集（覆盖所有 category 和 complexity）运行 MA-6D，产出最终验收报告。

### 5.3 跨模型评估

同一套测试用例可以跨多个模型运行，评估协议对不同 LLM 的普适亲和性。

## 6. 实现参考

评估模型已实现在 `eval/src/core/` 中：

| 模块 | 文件 | 职责 |
|------|------|------|
| 类型定义 | `types.ts` | `DimensionScores`, `LevelBreakdown`, `LearningCurveData`, `ConsistencyData` 等 |
| 分层验证 | `validator.ts` | L1-L4 四级递进验证，策略感知规则解析 |
| 6 维评分 | `scorer.ts` | `calcD1` ~ `calcD6` + `calcMA` 综合评分 |
| 测试运行 | `test-runner.ts` | 执行测试用例、管理重试、收集 I/O |
| 一致性检查 | `consistency-checker.ts` | 结构一致性和语义等价性的详细对比 |
| Prompt 构建 | `prompt/prompt-builder.ts` | 策略感知的 System/User Prompt |
| Few-shot 示例 | `prompt/few-shot-examples.ts` | 按 category 分组的可变数量示例 |
| 报告生成 | `report/report-generator.ts` | JSON + Markdown 格式报告 |
