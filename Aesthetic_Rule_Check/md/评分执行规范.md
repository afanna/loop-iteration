# 附录 D 评分执行规范（Scoring Pipeline Specification）

Version 2.0

---

# D.1 设计目标

评分执行规范用于定义：

HarmonyOS 卡片审美评分系统的整体执行流程。

规范所有模块的调用顺序。

保证：

- 流程固定
- 数据统一
- 可并行
- 可扩展
- 易维护

整个评分系统严格按照本规范执行。

---

# D.2 总体流程

评分系统执行流程如下：

```

        Query
          │
          ▼
         DSL
          │
          ▼
     Card Screenshot
          │
          ▼
─────────────────────────────

─────────────────────────────
          │
          ▼

          │
          ▼

       Step2

─────────────────────────────
          │
          ▼
     DSLContext
          │
          ▼
─────────────────────────────
       Step3

─────────────────────────────
          │
          ▼
```

---

# D.3 Step1：视觉检测

输入：

Screenshot。

输出：

VisionContext。

该步骤负责：

- OCR
- 图标检测
- 图片检测
- Button检测
- BoundingBox
- Color
- FontSize

所有后续模块：

不得再次OCR。

---

# D.4 Step2：DSL解析

输入：

DSL。

输出：

DSLContext。

负责：

提取：

- Key Text
- Entity
- Intent
- Importance

DSL：

仅用于：

信息完整性。

以及：

Intent Fitness。

禁止：

参与布局评分。

---

# D.5 Step3：Metric计算

所有Metric：

按照依赖关系执行。

---

## 第一组（可直接计算）

无需依赖其它Metric。

例如：

Whitespace

Margin

Spacing

Color

Contrast

Balance

Icon Ratio

Text Ratio

全部：

可并行。

---

## 第二组（依赖BBox）

需要：

VisionContext。

例如：

Grid

Alignment

Reading Order

Visual Weight

---

## 第三组（依赖DSL）

例如：

Information

Intent

Visual Priority

Reading Flow

必须等待：

DSLContext。

---

## 第四组（最终Metric）

例如：

Intent Fitness。

需要：

所有前置Metric。

全部完成。

---

# D.6 并行策略

建议：

采用：

ThreadPool。

例如：

```

Thread1

Whitespace

Thread2

Spacing

Thread3

Contrast

Thread4

Color

```

CPU：

多核利用。

减少：

整体耗时。

---

# D.7 Fusion

Fusion：

等待：

所有Metric完成。

输入：

```

MetricResult[]

```

输出：

```

OverallResult

```

禁止：

Metric之间互相修改Score。

---

# D.8 Explain生成

所有Metric完成以后。

统一生成：

Explain。

例如：

```

Whitespace

↓

当前值

↓

最佳值

↓

公式

↓

最终得分

```

Explain：

不得重新计算。

直接读取：

MetricResult。

---

# D.9 HTML生成

HTML：

读取：

OverallResult。

自动展示：

```

Overall

↓

Dimension

↓

Metric

↓

Explain

```

不得：

再次调用Metric。

---

输出：

result.json。

格式：

```json
{
    "overall":92.5,
    "confidence":0.94,
    "dimensions":{

        "layout":91,
```

CI。

数据库。

可视化。

# D.11 错误处理

任何一个Metric异常。

不得导致：

整个评分失败。

例如：

Contrast：

异常。

则：

```json
{
    "score":null,
    "status":"ERROR"
}
```

Fusion：

自动忽略。

同时：

降低：

Confidence。

---

# D.12 Cache

同一张截图：

VisionContext：

只计算一次。

所有Metric共享。

例如：

```

Screenshot

↓

OCR

↓

VisionContext

↓

20个Metric共享

```

避免：

重复OCR。

# D.16 设计原则

评分系统遵循以下原则：

1. 单一职责原则（SRP）
2. 开放封闭原则（OCP）
3. 配置优先原则（Configuration First）
4. 可解释原则（Explainable AI）
5. 连续评分原则（Continuous Scoring）
6. 美学优先原则（Aesthetic First）

其中：

**美学优先原则**为本系统最高原则。

所有评价指标必须围绕卡片视觉美学展开。

不得将功能正确性、DSL结构正确性等非美学因素纳入评分。

DSL 的唯一作用是辅助验证信息是否完整展示，以及辅助理解场景意图，不参与布局、视觉设计、一致性等美学维度评分。
