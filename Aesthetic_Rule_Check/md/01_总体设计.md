# 1. 项目背景

目前 HarmonyOS A2UI 已能够通过大模型自动生成 DSL，并完成卡片渲染。

整体流程如下：

```
Query
      │
      ▼
LLM生成DSL
      │
      ▼
DSL静态校验
      │
      ▼
HarmonyOS渲染
      │
      ▼
Card Screenshot
```

目前已经具备：

- DSL 自动生成
- DSL 自动校验
- HarmonyOS 自动渲染
- 自动截图

但是，目前缺少**卡片视觉质量（Visual Aesthetics）**的自动评价能力。

因此，需要建立一套：

> **HarmonyOS Card Visual Aesthetic Scoring System**

用于自动评价：

- 是否美观
- 是否符合设计规范
- 是否符合HarmonyOS卡片设计习惯
- 是否达到可交付质量

该系统将作为：

A2UI 自动生成流程最后一道质量门禁（Quality Gate）。

---

# 2. 项目目标

本项目目标不是检测 DSL。

不是检测 JSON。

不是检测代码。

也不是检测功能是否正确。

本系统仅评价：

> **最终渲染出来的卡片视觉效果。**

系统输入：

- Card Screenshot
- Query
- DSL（仅用于信息完整性）

系统输出：

- Overall Score（0~100）
- 五大维度评分
- 每个指标评分
- 可解释扣分原因
- HTML 可视化报告

---

# 3. 系统边界

本系统严格遵循以下边界。

## 3.1 Screenshot 是视觉评价唯一依据

布局、美学、颜色、层级、一致性全部来源于截图分析。

包括：

- Layout
- Color
- Typography
- Hierarchy
- Alignment
- Whitespace
- Visual Balance
- Reading Flow

全部仅分析截图。

不得读取 DSL 中的位置。

不得读取 DSL 中的颜色。

不得读取 DSL 中的字体。

因为：

系统评价的是：

> 用户最终看到的效果。

而不是：

设计者希望看到的效果。

---

## 3.2 DSL 仅用于信息完整性校验

DSL 不参与任何视觉评分。

DSL 仅作为：

Ground Truth（真实设计目标）。

用于回答一个问题：

> 应显示的信息是否最终成功显示？

DSL 不参与：

- 布局评分
- 色彩评分
- 对齐评分
- 层级评分
- 留白评分
- 美学评分

DSL 仅用于：

Information Coverage。

---

## 3.3 Query 仅用于识别卡片类型

Query 不参与评分。

Query 仅用于：

Intent Recognition。

例如：

```
今天北京天气
```

识别：

```
Weather Card
```

不同 Intent：

对应不同的设计目标。

例如：

天气卡：

温度应成为视觉焦点。

倒计时：

数字应成为视觉焦点。

音乐：

封面应成为视觉焦点。

因此：

Intent 不影响检测结果。

仅影响评分参数。

- 阴影

需要展示的信息。

OCR 对比。

计算：

Information Coverage。

## 5.3 Query

用于：

Intent 分类。

例如：

```
天气

会议

倒计时

音乐

健康
```

Intent 决定：

不同评分模型。

# 6. 五大评价维度

系统共包含五个一级评价维度。

| 一级维度 | 权重 | 数据来源 |
|----------|------|---------|
| 信息完整性 | 20 | DSL + OCR |
| 布局美学 | 25 | Screenshot |
| 视觉设计 | 30 | Screenshot |
| 设计一致性 | 15 | Screenshot |
| 场景契合度 | 10 | Screenshot + Intent |

最终：

```
Overall Score

=

Information

+

Layout

+

Visual

+

Consistency

+

Intent
```

总分：

100。

# 7. 美学评价原则

本系统采用：

Visual Metric

而不是：

Rule Engine。

即：

所有指标首先计算：

连续数学值。

例如：

```
Whitespace

=

0.42

Contrast

=

5.8

Focus

=

0.63

Balance

=

0.91
```

然后：

通过评分函数：

转换为：

```
Whitespace Score

93

Contrast Score

88

Focus Score

95
```

最后：

统一融合。

禁止：

大量：

```
if

padding<16

扣5
```

这种离散规则。

原因：

离散规则区分度低。

无法评价：

优秀设计。

---

# 8. 数学评分原则

所有 Metric 必须满足：

## 原则一

可计算。

必须具有明确数学定义。

禁止：

主观判断。

---

## 原则二

连续评分。

所有 Metric：

输出：

0~100。

不得：

仅输出：

PASS

FAIL。

---

## 原则三

可解释。

每个分数必须能够解释：

为什么：

92。

为什么：

81。

为什么：

65。

---

## 原则四

可复现。

同一截图：

不同时间：

评分一致。

不同机器：

评分一致。

---

# 9. Score Engine

每个 Metric：

计算：

```
Metric Score
```

每个维度：

融合：

```
Dimension Score
```

最后：

融合：

```
Overall Score
```

整个过程采用：

加权平均。

所有权重均可配置。

不得硬编码。

---

# 10. HTML 报告

系统最终生成：

HTML 可视化报告。

报告内容包括：

- Overall Score
- 五大维度得分
- 每个 Metric 得分
- 扣分原因
- 数学计算结果
- 截图展示
- OCR 结果
- DSL 信息覆盖率
- 可视化图表

所有扣分必须能够定位：

具体原因。

不得出现：

"AI认为不好看"

此类不可解释结果。

# 11. 开发原则

整个系统遵循：

## 模块独立

每一个 Metric：

一个 Python 模块。

互不依赖。

---

## 可扩展

新增 Metric：

无需修改已有模块。

仅注册即可。

---

## 可配置

所有参数：

均写入配置文件。

包括：

- 权重
- 最佳值
- σ
- 阈值
- Intent Profile

不得写死。

---

## 可测试

每一个 Metric：

必须拥有：

单元测试。

能够独立验证数学计算正确性。
